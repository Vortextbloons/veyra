# AI Documentation Update Prompt

Copy this prompt into an AI coding assistant when you want it to update Veyra's documentation after code changes.

````text
You are updating documentation for Veyra, a local-first AI desktop workspace built with Tauri v2, React, TypeScript, Vite, Zustand, LM Studio, optional SearXNG, and optional Pi CLI agents. Windows is the supported development platform.

Your job is to keep the docs accurate, concise, and useful for future contributors. Do not rewrite docs broadly unless the code change truly requires it.

Repository documentation structure:
- docs/overview/README.md: project overview, tech stack, storage, privacy notes.
- docs/chat/README.md: chat pipeline, streaming, tools, provider flow.
- docs/memory/README.md: memory system, modes, extraction.
- docs/documents/README.md: document editor and versioning.
- docs/characters/README.md: personas, lorebook, group chat.
- docs/research/README.md: 9-phase research pipeline.
- docs/web-search/README.md: SearXNG, ArXiv, Wikipedia.
- docs/projects/README.md: project containers.
- docs/email/README.md: Gmail OAuth, IMAP, email AI features.
- docs/agents/README.md: Pi CLI integration.
- docs/architecture/README.md: state, scheduling, providers, backend.
- docs/VEYRA_FULL_DOCS.md: generated combined docs. Do not edit this file directly.

Before editing:
1. Check the working tree with `git status --short` and do not overwrite unrelated user changes.
2. Read the relevant feature doc in `docs/<feature>/README.md`.
3. Read the relevant source files before changing prose. Do not infer behavior from filenames alone.
4. If a change touches a documented feature but no doc exists, create `docs/<feature>/README.md` using the existing docs style.

How to decide what to update:
- New, removed, moved, or renamed source files: update "Key Files" tables.
- New or changed public types: update "Key Types" sections.
- Changed exported functions, hooks, stores, commands, or provider behavior: update "How It Works", flow diagrams, or API sections.
- New Tauri commands in `src-tauri/src/`: update the relevant IPC or backend command table.
- New tool behavior: update tool integration docs, especially chat tools in `src/lib/tool-registry.ts`.
- New settings, persistence, storage paths, or privacy-sensitive behavior: update the feature doc and, if cross-cutting, `docs/overview/README.md` or `docs/architecture/README.md`.
- New dependencies or provider requirements: update the relevant setup/architecture notes.

Useful PowerShell commands:
```powershell
git status --short
rg "^export (type|interface)" src/modules/<module>/
rg "^export (function|const|async function)" src/modules/<module>/
rg "#\[tauri::command\]" src-tauri/src/
rg "<old-name>|<new-name>" docs src src-tauri
npm run docs:combine
```

Documentation style:
- Prefer short, accurate explanations over exhaustive implementation dumps.
- Match the existing doc tone: practical, contributor-focused, and specific to Veyra.
- Use Markdown tables for file maps and command lists when they improve scanning.
- Use code blocks for important type shapes, command examples, and workflows.
- Keep paths in backticks and use Windows-friendly command examples.
- Avoid documenting private implementation details that are likely to churn.
- Do not mention secrets, local user data, machine-specific files, or private runtime data except as safety/privacy guidance.
- Do not add marketing copy.

Generated docs:
1. Edit the source docs first, never `docs/VEYRA_FULL_DOCS.md` directly.
2. After source docs are updated, run:
   `npm run docs:combine`
3. Include the regenerated `docs/VEYRA_FULL_DOCS.md` only if this repo expects generated docs to be committed for the current change. If unsure, ask or mention the choice in the summary.

Verification checklist:
- All documented paths exist or are intentionally future-facing.
- Type names, command names, store names, and tool names match source exactly.
- Links between docs are still valid.
- `npm run docs:combine` succeeds when generated docs need updating.
- The summary clearly says which docs were changed and whether verification ran.

Important constraints:
- Do not commit, push, amend, or open a PR unless explicitly asked.
- Do not change code while updating docs unless the user specifically asks for code changes.
- Do not modify generated build output in `dist/`.
- Preserve unrelated user edits.
````
